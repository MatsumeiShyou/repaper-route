# Fast-Path (バイパス) プロトコル要件定義

## 1. 目的表示
Double Loop Governance や SADAテスト などの厳格な統治機構は、システムの品質と安全性を確保・進化させる重要な基盤です。しかしながら、すべての微細な変更（例: ボタンのラベル変更等）に対してフルセットの制約チェックやテスト構築を実施することは、実践的な開発ベロシティを著しく低下させます。
本プロトコルは、システムの安定性に影響を与えない「極めて軽微な変更」に限定して、検証プロセスを合理化（Fast-Path）するための明確な基準と手順を定義します。

## 2. 適用基準 (Criteria)
以下の**すべて**を満たす変更のみ、Fast-Pathの対象とすることができます。

1. **表示層のみの変更**: UIの文言修正、色・スタイル（CSS/Tailwind）の微小な変更、静的コンテンツの差し替えであること。
2. **ロジックの非介在**: 状態管理（`useState`, `useEffect` 等）や通信ロジック（API, RPC呼出）、DBスキーマ、ビジネスロジックに一切変更を加えないこと。
3. **テスト要件の非破壊**: 既存のSADAテスト、及びその他のCIテスト環境を破壊しない変更であること。

## 3. Fast-Path における検証スキップの免除範囲
Fast-Path が適用される場合、以下のプロセスを合法的にスキップ、または簡略化できます。

- **SADA テストの新規作成免除**: 当該UIのための新たなSADAテストの記述をパスできます（ただし、既存テストが通ることは必須要件）。
- **SVP物理ロックの論理解除の簡略化**: `[Audit: Fast-Path 適用 - UI文言整理のため]` のような簡易的な理由での一括コミット進行が許可されます。
- **Pre-flight / Seal のバイパス**: Pre-flight自体の実行は必須ですが、当変更と無関係の部分で警告（Warning）が発生している場合、早期警告を無視して進行可能です。

## 4. 運用ルールの制限事項
- Agentが自己判断で密かに Fast-Path を「実行」してはなりません。ただし、適用条件を満たすと判断した場合は、**ユーザーに対して積極的に Fast-Path の適用を「提案」することが憲法上（Creative Option等に準じて）推奨**されます。
- Agentが「本件はFast-Pathが適用可能です（SADAテスト省略等を提案）」とユーザーに申請し、ユーザーが承認（`y`など）した場合にのみ、正式に適用手順（Audit記述等）を実行します。
- もし Fast-Path を適用した変更によって不具合（リグレッション等）が発生した場合、その変更はただちに Roleback され、原因が解明されるまで該当コンポーネントに対する Fast-Path の適用権限は凍結されます。
