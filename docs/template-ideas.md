# テンプレート機能 設計評価 & アイデアログ
> 記録日: 2026-03-23

---

## 1. テンプレートの本質

テンプレートは単なる「ルート保存」ではなく、**「繰り返しの暗黙知（属人的ノウハウ）を形式知に変換する装置」**。

---

## 2. 現行設計の評価

### ✅ 強み
- **骨格データ（Skeleton）思想**: ドライバー・時刻を除外し「仕事の純粋な中身」のみ保存 → 日々の欠員・車両変更に強い
- **マスタ不整合ガード**: 削除済み顧客を`unassigned`へ安全退避する設計
- **グリーディ展開の決定論性**: `visit_slot → customer_id` 優先度が固定 → 同入力から常に同結果（説明可能）
- **Ghost機能**: 空白日に過去4週同曜日の実績を薄く表示し、認知負荷を軽減

### ⚠️ 改善余地
| # | 課題 | 優先度 |
|---|---|---|
| 1 | 展開結果の「なぜ退避されたか」がUIに見えない | 🔴 高 |
| 2 | `absentCount`（欠員想定数）がExpanderに未連携（UIと挙動が不一致） | 🔴 高 |
| 3 | `required_vehicle \|\| 'AT'` フォールバックが暗黙化されており誤配車リスクあり | 🟡 中 |
| 4 | 複数テンプレートがある場合、自動選択の上書き変更手段がない | 🟡 中 |
| 5 | Ghost表示のオン/オフ制御がない | 🟢 低 |

## 3. 新アイデア

### 💡 アイデア① 暗黙知マイニング（**最推奨**）
**実装コスト: 低 / UX価値: 🔥🔥🔥**

過去の実績データを自動分析し、「これテンプレート化しませんか？」と能動的に提案する。

```
「過去3ヶ月の月曜日を分析した結果、
 80%以上同じ構成が使われています。
 → テンプレート化を推奨します [保存する]」
```

- **既存資産の活用**: `routes`テーブルの蓄積データをそのまま使用可能
- **Ghost機能の発展版**として設計できる

---

### 💡 アイデア② テンプレートのブレンド（合成配車）
**実装コスト: 中 / UX価値: 🔥🔥**

複数テンプレートを「配合率」で合成する機能。

```
[月曜基本ルート 70%] + [繁忙期補強ルート 30%]
  → 重複案件は優先度高い方を採用
  → 不足案件は補強ルートから補充
```

- **ユースケース**: お盆・年末など、通常ルートに特殊案件を追加したい日。現状は特殊テンプレートを1から作るしかない。

---

### 💡 アイデア③ テンプレート差分の可視化（**最推奨**）
**実装コスト: 中 / UX価値: 🔥🔥**

テンプレート適用時のスナップショットを保存し、当日の最終配車との差分を自動記録・レポートする。

```
📊 本日の乖離レポート
  ✅ テンプレート通り: 12件
  🔄 時間を変更:        3件（A, C, F案件）
  ➕ 追加:              2件
  ➖ 退避:              1件（マスタ削除）
  → 乖離率: 30%（先月比 +5%）
```

- **ユースケース**: ルートが非効率化した原因分析、テンプレート自体の改善トリガー

---

### 💡 アイデア④ コンテキスト感知テンプレート
**実装コスト: 高 / UX価値: 🔥🔥🔥（将来投資）**

曜日・週番号以外の外部コンテキストでテンプレートを切り替える。

| コンテキスト | 活用例 |
|---|---|
| 天候 | 雨の日は遠方エリアを後回しにするルート |
| 出勤人数 | 欠員時専用のコンパクトルート |
| 特殊フラグ | 月末・祝日前日・繁忙期 |

---

### 💡 アイデア⑤ ドライバー固有テンプレート（パーソナライズ配車）
**実装コスト: 中 / UX価値: 🔥**

各ドライバーの「得意ルート・担当エリア」をドライバーIDに紐付け、展開時に優先マッチングする。

```
山田さん → 北エリアの案件を優先
佐藤さん → 大型車両が必要な案件を優先
```

- **視点**: 属人性を「排除」ではなく「最大化」するアプローチ

## 4. テンプレートのライフサイクル管理（追加 ↔ 削除の対称設計）

> **「追加できるなら削除もできる。追加と削除は対称に設計すべき。」**

テンプレートが増えていくことで「使えないゴミ」も増える。
**常に今使えるものだけが見える状態**を維持するために、ライフサイクル管理が不可欠。

### テンプレートの状態遷移

```
[下書き] → [有効] → [アーカイブ] → [完全削除]
             ↑           ↓
           (復元)      (参照のみ可)
```

### 各状態の定義

| 状態 | 意味 | UIでの扱い |
|---|---|---|
| **下書き** | まだ使わない / 検討中 | ダッシュボードに薄く表示 |
| **有効** | 通常使用中 | ダッシュボードに表示・適用可能 |
| **アーカイブ** | 廃止だが履歴として保存 | 非表示（検索すれば参照可） |
| **完全削除** | 物理削除 | 管理者権限のみ・要二重確認 |

### 「アーカイブ」を「削除」より優先すべき理由

- 過去の乖離レポート（§8）が参照できなくなる
- 「なぜこのルートをやめたか」の意思決定が失われる
- 誤操作での回復ができない

→ **デフォルトはアーカイブ**。完全削除は管理者が明示的に行う二段階方式を推奨。

### 自動アーカイブ候補の検知

```
💡 以下のテンプレートが長期間未使用です：
  「繁忙期補強ルート」→ 最終使用: 8ヶ月前
  「旧北エリアルート」→ 最終使用: 1年以上前
  → アーカイブしますか？ [する] [このまま残す]
```

## 6. テンプレート選択の決定変数ピラミッド

テンプレートを選ぶ際の変数には優先順位がある。

| 順位 | 変数 | 意味 | データ取得元 |
|---|---|---|---|
| ① | 曜日 × 第N週 | 仕事の種類と量 | 日付から自動算出（実装済み） |
| ② | 出勤人数・欠員数 | コース本数・積載容量 | アプリ内データ |
| ③ | スポット案件数 | 動的な余白の消費 | アプリ内データ |
| ④ | 天候・道路状況 | 所要時間の膨張 | 外部API（将来） |
| ⑤ | 顧客側事情（休業等） | 案件の自然消滅 | マスタデータ |

①〜③は**アプリ内データのみ**で実現可能。④以降は外部コンテキスト。

### ②×③の関係

スポット案件が多い日は「人員が足りていても実質的に人員不足と同じ」。
→ テンプレートには**あえて余白（バッファ）**を設けて設計する考え方が重要。

```
欠員0名 × スポット0件 → フルテンプレート展開
欠員0名 × スポット多発 → コンパクトテンプレート（余白あり）
欠員1名 × スポット0件 → 案件を他コースに分配
```

## 7. 過去パターン活用と「鮮度設計」

事前にテンプレートを全パターン用意するのは非現実的（例: 5曜日×5週×3欠員=75通り）。
→ **過去の実績データをテンプレートの代替として使う**。

### 鮮度ゾーン
```
直近 1〜3週  ← 鮮度 ◎  ただし偶発事象を含みやすい
直近 4〜8週  ← 鮮度 ○  ゴールデンゾーン（Ghost機能の現状参照範囲）
直近 3〜6ヶ月 ← 鮮度 △  季節変動を含む
1年以上前    ← 鮮度 ×  人員・顧客が変わっている可能性大
```

### 加重平均パターン
複数週を重み付き合成することで、偶発事象を自然に薄める。

```
今日の提案 =
  先週同曜日   × 0.4
+ 2週前同曜日  × 0.3
+ 3週前同曜日  × 0.2
+ 4週前同曜日  × 0.1
```

### 陳腐化検知
```
⚠️ このテンプレートは3ヶ月更新されていません。
   直近4週の実績と30%以上乖離しています。
   → 実績から再生成しますか？ [はい] [このまま使う]
```

## 8. 設計哲学（Design Philosophy）

> **「テンプレートは編集可能な土台。そこから人間と機械が協働して、その日の最大効率・最大生産性を実現する。」**

### 役割の3層構造

```
Layer 3: 最適化レイヤー    ← 効率・生産性の追求（将来目標）
              ↑ 微調整・承認
Layer 2: 人間の判断レイヤー ← 最終決定・例外対応
              ↑ 変数を注入
Layer 1: テンプレート（土台） ← 編集可能なパターン（現在地）
```

### テンプレートと最適化の違い

| | テンプレート | 配車最適化 |
|---|---|---|
| **問い** | 「先週と同じか？」 | 「今日の最大効率は何か？」 |
| **視点** | 過去→現在（再現） | 現在→未来（最良解） |
| **出力** | 「いつものやつ」 | 「今日のベスト」 |

テンプレートは「仮説」、最適化が「検証と確定」。

## 9. 配車の究極目標

> **「今日の人員・車両・時間帯の制約の中で、回収量を最大にする最も少ない投入資源の組み合わせは何か？」**

### 生産性の2軸
```
軸①: 量の最大化 → 1日に回収できる件数・重量を最大に
軸②: 資源の最小化 → 同じ量を、より少ない人員・時間・燃料で

最高効率 = 同じ回収量を、より少ない投入で実現すること
```

### 理想のアーキテクチャ（まとめ）

```
事前テンプレート（骨格・意図を込めた設計）
           +
過去実績パターン（直近4〜8週の加重平均）
           ↓
    両者を比較して「より信頼できる方」を提案
    乖離が大きければ警告 → ユーザーが判断
           ↓
    最適化エンジンが今日の変数を注入
    （人員数 × スポット数 × 車両状態）
           ↓
    「今日のベスト配車案」を提示
           ↓
    人間が最終確認・微調整 → 確定
```

## 10. 「持ち回り配車」のモラルハザード対策（ゲーミフィケーション）

> **「テンプレート展開後に何も考えずに保存する（サボる）」ことを防ぎ、効率化のモチベーションを設計する。**

本アプリは属人性を排除し「誰でも配車できる持ち回り制」を目指している。しかし、共有責任は「誰も最適化に責任を持たない（適当に組む）」というモラルの低下を生みやすい。
**AIによる監視ではなく、決定論的スコアリングとUIの可視化による牽制**でこれを解決する。

### 10.1. 決定論的スコアリング（ルールの透明化）
配車盤の「効率スコア」を明確な数式で毎朝計算する。ブラックボックスを排除し、誰もが納得できる計算式とする。

**計算式（例）：**
* **基本点**: (積み込んだ総案件数 / 出勤人数) × 定数
* **加点要素**: 
  * 終了見込み時間が全コースで均等（バラツキが少ない）= ＋10点
  * スポット案件をうまく吸収できた = ＋5点/件
* **減点要素（サボり検知）**: 
  * 不自然な空白がある（未配車エリアへの意図的な放置）= −20点/件
  * 特定のドライバー（例：新人）にだけ過剰に案件が集まっている = −15点

### 10.2. UIでのリアルタイム・フィードバック
配車を組んでいる最中（`BoardCanvas`）に、右上に**現在の配車スコア**をゲームのようにリアルタイム表示する。

```text
📊 本日の配車効率スコア: 85点 / 100点 (評価: A)
   [+ 加点] 均等な割り振り (+10) 
   [- 減点] 未配車が2件残っています (-20)
   👤 責任者: 山田太郎
```

* **牽制効果**: 保存ボタンを押す前に「自分は今70点の適当な配車をしようとしている」という自覚を持たせる。
* **承認欲求刺激**: 「95点」を出した日はUI上で称賛演出（色が変わる等）を行い、達成感を与える。

### 10.3. 人事評価・査定への連動（Accountability）
確定した配車スコアを、そのままデータベースに記録し、人事評価の定量的データとして活用する。

* **責任者の明確化**: 最後に「確定ボタン」を押したユーザーID（`locked_by`）にその日のスコアが紐付く。
* **データ活用**: 月末に「当番ごとの平均スコア」を集計。人員不足の中で高効率ルートを組み立てた優秀な配車担当者には、正当なボーナス査定（インセンティブ）を与える根拠とする。

テンプレートは「手足を抜くためのツール」ではなく、「高得点を出すための土台」として再定義される。
