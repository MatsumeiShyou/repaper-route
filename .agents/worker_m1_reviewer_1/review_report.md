## Quality Review Findings

### [Major] Finding 1: Incorrect Nth Weekday Recurrence Matching in `PeriodicJobImporter`
- **What**: The logic to match Nth weekdays (such as `mon1` for the first Monday) in `collection_days` is incorrect.
- **Where**: `C:\Users\shiyo\開発中APP\RePaper Route\apps\repaper-route\src\lib\PeriodicJobImporter.ts` lines 36-69
- **Why**: 
  - For the **Object format** (which is the default DB format generated by `serializeMasterData`), if `collection_days` is `{ mon1: true, mon: false }` (1st Monday), `isDayMatch` checks `!!daysObj['mon']` which evaluates to `false`. As a result, the job is never imported on any Monday, including the 1st Monday.
  - For the **Array format**, if `collection_days` is `['mon1']`, `d.toLowerCase().startsWith('mon')` evaluates to `true` on any Monday (e.g. 1st, 2nd, 3rd Mondays). Since `p.recurrence_pattern` is null, the job is imported on every Monday, disregarding the week restriction.
- **Suggestion**: 
  - Update the Object case to also check for Nth week day keys:
    `isDayMatch = !!daysObj[dayKey] || !!daysObj[\`${dayKey}${nth}\`]`
  - Update the Array case to only match exact matches or matching Nth week day strings:
    ```typescript
    isDayMatch = collectionDays.some(d => {
        if (typeof d !== 'string') return false;
        const lower = d.toLowerCase();
        return lower === dayKey || lower === `${dayKey}${nth}`;
    });
    ```

### [Major] Finding 2: Robustness Issue in `cleansePurgedFields` (Destructive Object Cloning)
- **What**: Destructures non-plain objects such as `Date` or `RegExp` into empty plain objects `{}`.
- **Where**: `C:\Users\shiyo\開発中APP\RePaper Route\apps\repaper-route\src\utils\serialization.ts` lines 110-132
- **Why**: `typeof date` evaluates to `'object'`. Using `{ ...data }` on Date/RegExp objects copies only enumerable properties, resulting in `{}`. This completely destroys these objects.
- **Suggestion**: Ensure that only plain objects are recursively cloned:
  ```typescript
  const isPlainObject = Object.prototype.toString.call(data) === '[object Object]';
  if (!isPlainObject) return data;
  ```

### [Minor] Finding 3: Potential Crash on Unexpected API payload in `MasterDataContext`
- **What**: `dRes.data` is mapped without checking if it is an array.
- **Where**: `C:\Users\shiyo\開発中APP\RePaper Route\apps\repaper-route\src\contexts\MasterDataContext.tsx` line 42
- **Why**: If the API returns an object (e.g. error payload) instead of an array, `(dRes.data || [])` will evaluate to that object, and calling `.map` on it will throw a TypeError.
- **Suggestion**: Use `Array.isArray(dRes.data) ? dRes.data : []` before mapping.

### [Minor] Finding 4: Case-sensitive Regex for Day Keys in `normalizeDays`
- **What**: Regex for Nth weekday keys is case-sensitive.
- **Where**: `C:\Users\shiyo\開発中APP\RePaper Route\apps\repaper-route\src\utils\serialization.ts` line 92
- **Why**: The regex `/^[a-z]{3}[1-5]$/` expects lowercase keys. If any legacy data or third-party payload has uppercase keys like `'Mon1'`, they will be ignored.
- **Suggestion**: Make the regex case-insensitive (`/^[a-z]{3}[1-5]$/i`).

## Adversarial Review & Stress-Test Results

**Overall risk assessment**: MEDIUM

### Challenge 1: Bypass of Token Refresh due to Direct LocalStorage Extraction
- **Assumption challenged**: Reading auth token directly from `localStorage` assumes that the stored token is always valid and does not need refresh.
- **Attack scenario**: If the user's access token in `localStorage` has expired, but a valid refresh token exists, calling `nativeSupabaseFetch` bypasses Supabase's auto-refresh mechanism. This results in a 401 Unauthorized API error, whereas a call through standard Supabase Client SDK would have silently refreshed the token.
- **Blast radius**: API requests will fail with 401 until the page is reloaded or the user logs in again.
- **Mitigation**: Implement a retry handler on 401 errors that invokes `supabase.auth.getSession()` to trigger a token refresh, then retries the native fetch.
